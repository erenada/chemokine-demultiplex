# Chemokine Receptor Barcode Quantification — Multi-Sample Pipeline

**Author:** Eren Ada, PhD  
**Pipeline version:** 2.0

---

## Overview

This pipeline quantifies chemokine receptor barcodes across multiple pre-demultiplexed FASTQ files. It is designed for experiments where each sample has already been sequenced into a separate FASTQ file (e.g., standard Illumina library preparation with per-sample indices). The pipeline identifies and counts barcode occurrences using approximate string matching, handles both forward and reverse-complement orientations, and outputs raw and normalized count matrices alongside quality control reports.

**This pipeline covers two phases:**

| Phase | Script | Purpose |
|---|---|---|
| Phase 1 | `phase1_data_preparation_multi.py` | Input validation, QC assessment across all files, reverse-complement barcode generation |
| Phase 3 | `phase3_barcode_quantification_multi.py` | Barcode quantification across all samples, count matrix construction, normalization |

Phase 2 (sample demultiplexing) is intentionally skipped because input files are already per-sample.

---

## When to Use This Pipeline

Use this pipeline when:
- Each sample is in its own FASTQ file (pre-demultiplexed).
- You want to quantify a defined set of chemokine receptor barcodes across multiple conditions and replicates.
- Your experiment has multiple conditions (e.g., positive selection vs. negative control) with replicates.

For experiments where all samples are pooled in a single FASTQ file and need to be demultiplexed first, use the main `scripts/` pipeline instead.

## Input Files

### 1. Sample mapping file

A CSV file that maps each FASTQ file to its experimental metadata.

**Required columns (case-sensitive):**

| Column | Description |
|---|---|
| `Sample_no` | Integer sample index |
| `Sample_annotation` | Unique sample name used in output files |
| `Condition` | Experimental condition label (e.g., `GFP`, `Neg`) |
| `Replicate` | Replicate number |
| `fastq_file` | Filename of the corresponding FASTQ file |

**Example:**

```csv
Sample_no,Sample_annotation,Condition,Replicate,fastq_file
1,Sample-GFP-Rep1,GFP,1,Sample1_GFP.fastq
2,Sample-GFP-Rep2,GFP,2,Sample2_GFP.fastq
3,Sample-GFP-Rep3,GFP,3,Sample3_GFP.fastq
4,Sample-Neg-Rep1,Neg,1,Sample4_Neg.fastq
5,Sample-Neg-Rep2,Neg,2,Sample5_Neg.fastq
6,Sample-Neg-Rep3,Neg,3,Sample6_Neg.fastq
```

The `fastq_file` values must exactly match the filenames present in your FASTQ directory.

### 2. Chemokine barcode file

A headerless two-column CSV with chemokine receptor names and their corresponding barcode sequences.

**Format:** `ChemokineName,BarcodeSequence`

**Requirements:**
- No header row.
- Barcodes must contain only A, T, C, G characters.
- Expected barcode length: 20 bp.

**Example:**

```csv
CCR1,ATGCGTACGTTAGCTAGCTA
CCR2,CGTAGCTAGCTTAGCTACGA
CCR3,GCTAGCTAGGCTAGCTAGCT
```

### 3. FASTQ files

- One FASTQ file per sample, uncompressed (`.fastq`). If gzipped, decompress before running.
- All files must be placed in a single directory.
- Filenames must match the `fastq_file` column in the sample mapping file exactly.

## Configuration

Before running, open each script and update the file paths in the `main()` function at the bottom to match your project layout:

**`phase1_data_preparation_multi.py`:**
```python
sample_mapping     = "path/to/reference_files/sample_mapping.csv"
chemokine_barcodes = "path/to/barcodes.csv"
fastq_dir          = "path/to/fastq/"
output_base_dir    = "path/to/output/"
```

**`phase3_barcode_quantification_multi.py`:**
```python
sample_mapping  = "path/to/reference_files/sample_mapping.csv"
barcode_file    = "path/to/output/reference_files/chemokine_barcodes_with_rc.csv"
fastq_dir       = "path/to/fastq/"
output_base_dir = "path/to/output/"
```

Note: `barcode_file` in Phase 3 points to `chemokine_barcodes_with_rc.csv`, which is generated by Phase 1. Set it to wherever Phase 1 wrote its output.

### Sensitivity parameters (Phase 3)

| Parameter | Default | Description |
|---|---|---|
| `max_mismatches` | `2` | Maximum edit distance allowed when matching a barcode to a read. Increase to be more permissive; decrease for stricter matching. |
| `min_quality` | `20` | Minimum average Phred quality score per read. Reads below this threshold are excluded from quantification. |

## How to Run

Run from the repository root. Phase 1 must be completed before Phase 3.

**Step 1 — Data preparation and QC:**

```bash
python3 multi_sample/scripts/phase1_data_preparation_multi.py
```

Review the QC output at `qc_reports/phase1_multi_file_assessment.png` before proceeding.

**Step 2 — Barcode quantification:**

```bash
python3 multi_sample/scripts/phase3_barcode_quantification_multi.py
```

## How Barcode Matching Works

Phase 3 uses approximate string matching via [edlib](https://github.com/Martinsos/edlib) to locate barcodes within reads:

1. Both the forward barcode and its reverse complement are searched in every read.
2. A match is accepted if the edit distance is within `max_mismatches`.
3. If a read contains matches to more than one chemokine, each is counted independently using the best-scoring hit per chemokine.
4. Ties (identical score for two match locations) are discarded to avoid ambiguous assignments.
5. Reads below the `min_quality` Phred threshold are skipped entirely.

## Troubleshooting

| Issue | Likely cause | Solution |
|---|---|---|
| `FileNotFoundError` on the barcode file in Phase 3 | Phase 1 has not been run yet | Run Phase 1 first; it generates `chemokine_barcodes_with_rc.csv` |
| `KeyError` on sample mapping columns | Column names do not match exactly | Verify column names are `Sample_no`, `Sample_annotation`, `Condition`, `Replicate`, `fastq_file` (case-sensitive) |
| FASTQ format error on load | Gzipped or corrupted file | Decompress FASTQ files before running (`gunzip *.fastq.gz`) |
| Very low detection rates | Barcode mismatch too strict, or wrong barcode file | Check that barcodes match the library design; consider increasing `max_mismatches` to 3 |
| `IndexError` writing `.fxi` index files | Write permissions missing | Ensure the FASTQ directory is writable; `pyfastx` writes index files alongside each FASTQ |
